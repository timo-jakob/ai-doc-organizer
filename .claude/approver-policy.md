# Claude Approver — Policy

This file is the **source of truth** for the Approver's per-PR-type
criteria. The agent reads it on every run; changes to this file ship
through normal code review. A policy-change PR is evaluated by the
*previous* version of this policy.

It has two marked parts, rendered into this one file by bootstrap:

- **Core** (this part) — the language-independent judgment criteria:
  type detection, baseline criteria, test representativeness, per-type
  must-haves and risk factors, confidence calibration, and the verdict
  protocol.
- **Language overlay** (appended below) — the python-specific
  fluency: which paths mean what in this repo, which tools verify API
  stability, runtime-bump surfaces, test idioms, and the triage-agent
  vocabulary. Where the core says "per the language overlay", that
  section is the reference.

**How the policy is applied (epic #476):** the Approver is
**user-invoked locally** via `/development-python:approve <PR>`
— not by a GitHub Actions workflow. The skill mints an Approver App
token from the local Keychain, runs the same `python-approver`
agent CI used to run, and posts the verdict as `claude-approver-bot`.
This file is what that run consults: **it defines what the skill
considers "approvable"** — the type detection, baseline criteria,
per-type must-haves, and risk factors below are the entire decision
surface. Editing this file changes what gets approved; nothing
server-side overrides it.

Bootstrap-generated by `/development:bootstrap --claude-approver true`.
See `development/skills/bootstrap/docs/CLAUDE-APPS.md` and
`development/skills/bootstrap/docs/APPROVER-APP.md` in the plugin
family for the design and the broader Approver story.

## How the Approver reads this file

For every PR the approve skill is invoked on:

1. Determine PR type per **Type detection** (below).
2. Apply **Baseline criteria** (every type).
3. Apply **Test representativeness** to every added/changed test.
4. Apply the **Per-type must-have criteria** for the detected type —
   the core section first, then the language overlay's additions for
   the same type.
5. Build a risk register from the **Per-type risk factors** (core +
   overlay), folding in the **Review dossier** if the PR body carries one.
6. Calibrate confidence per **Confidence calibration** (the overlay may
   add caps).
7. Render the verdict: `APPROVE` / `REQUEST_CHANGES` / `COMMENT`-with-reservations.

If type cannot be unambiguously determined, the Approver emits an
"ambiguous type" finding and `REQUEST_CHANGES`.

## Type detection

### Primary: conventional-commit prefix in the PR title

| Prefix (with or without scope) | Type |
|---|---|
| `feat:` / `feat(*):` | `feat` |
| `fix:` / `fix(*):` | `fix` |
| `refactor:` / `refactor(*):` | `refactor` |
| `chore(deps):` / `chore(deps:*)` | `chore_deps` |
| `chore(deps-major):` | `chore_deps_major` |
| `chore(runtime):` | `chore_runtime` |
| `security:` / `security(*):` | `security` |
| `docs:` / `docs(*):` | `docs` |
| `test:` / `test(*):` | `test` |
| `ci:` / `ci(*):` / `build:` / `build(*):` | `ci` |
| `chore:` / `chore(*):` *(excluding deps/runtime above)* | `chore` |
| `revert:` / `revert(*):` | `revert` |
| `hotfix:` / `hotfix(*):` | `hotfix` |

A `!` after the type or scope (`feat!:`, `fix(api)!:`) signals a
breaking change. The Approver treats this as a per-type flag, not a
separate type.

### Fallback: diff heuristic (when title prefix is missing)

- Diff entirely under documentation paths (`docs/`, `*.md`, `LICENSE`,
  `CONTRIBUTING.md`, `README.md`, `SETUP.md`) → `docs`.
- Diff entirely under the language's **test paths** (per the language
  overlay) → `test`.
- Diff entirely under workflow / pre-commit / **build-config paths**
  (`.github/workflows/`, `.pre-commit-config.yaml`, plus the language
  overlay's build-config set) → `ci`.
- Only the language's **dependency manifests** changed (per the
  language overlay) → `chore_deps`. If the language's **runtime pin**
  (per the overlay — e.g. a `Dockerfile` FROM line or a toolchain
  version) is the only non-manifest diff, → `chore_runtime`.
- Otherwise: **ambiguous** — emit a finding.

### Tiebreaker: author hint (when primary and fallback disagree)

- `dependabot[bot]` → `chore_deps`, unless title declares `chore(deps-major):`.
- `renovate[bot]` → `chore_deps`, unless title declares `chore(deps-major):`
  or the bump is major per the version delta in the title.
- `snyk-bot` → `security`.
- `github-actions[bot]` → `ci`.
- `claude-maintenance[bot]` → whichever the title prefix says; the
  maintenance pipeline always sets a correct conventional prefix.
- Anyone else: **still ambiguous** — emit a finding.

## Baseline criteria (apply to every type)

1. **CI is green at the head SHA.** The approve skill verifies this
   before evaluating and re-checks it at evaluation time (HEAD SHA
   freeze) — there is no server-side workflow gate to rely on.
2. **No new findings introduced by this PR** in the repo's static-
   analysis and security scanners (SonarCloud/SonarQube, Snyk, CodeQL
   alerts). "Introduced" = present in the PR's head but not in the
   merge-base.
3. **PR description has `## Type`, `## Summary`, and `## Test plan`
   sections.** Missing fields are finding-level.

   **Vendor-bot exception — exact allowlist, nobody else.** PRs
   authored by `dependabot[bot]`, `renovate[bot]`, or `snyk-bot`
   (`gh` may render the App logins as `app/dependabot` /
   `app/renovate`) satisfy this criterion with their native
   machine-generated body: those bots cannot emit the repo's PR
   template, and their body carries the equivalent evidence
   (package, version delta, compatibility data, changelog links).
   Derive `Type` from the conventional-commit title prefix. This
   exception is those three logins ONLY — any other bot, GitHub
   App, or human author is held to the full criterion; an unknown
   App opening template-less PRs is a case for human judgment (emit
   the finding as usual), never an automatic pass.
4. **No conflict markers in the diff.** A literal `<<<<<<< HEAD` in any
   tracked file is a critical finding (`REQUEST_CHANGES` regardless of
   type).
5. **No new `TODO` / `FIXME` without a tracking issue link.** A bare
   `TODO: fix this` is a finding; `TODO(#123): ...` is fine.
6. **No new secrets in the diff.** The scanners should already catch
   this; defence-in-depth re-check at this layer.
7. **No new dependency without compatibility-score evidence.** For
   dep-bumps, the PR body should contain the Dependabot compatibility
   score, Renovate's age/confidence badges, or a release-notes
   excerpt; absence is a finding.

## Test representativeness (applies to every added/changed test)

Coverage percentage is **necessary but not sufficient**. A PR can move
the coverage number without verifying any behaviour; the Approver's job
is to catch exactly that. Three lenses:

1. **Assertions bind to behaviour.** Flag coverage padding: assert-true
   filler, assertions only on stub/mock return values (that verifies
   the mock, not the code), tests that mock the very unit they test,
   test names promising behaviour the assertions never check, and
   left-in "TODO: write test" bodies. The language overlay lists the
   concrete idioms to look for.
2. **The test pyramid holds.** New behaviour needs **unit tests** for
   its logic *and* — when the change crosses a component boundary
   (I/O, a framework layer, another service, the CLI surface) — an
   **integration-level test** exercising that boundary. A feature
   verified only by unit tests against mocks of the boundary it exists
   to serve is under-tested: emit a finding naming the untested
   boundary.
3. **Interface-level test evidence.** When the PR changes a public
   interface (API endpoint, CLI flag, exported symbol, event schema),
   look for test evidence **at that interface** — a test that drives
   the interface as a consumer would, not just the internals behind
   it. *Forward reference:* the acceptance-test stage of the epic this
   policy belongs to will make such evidence a machine-checked
   artifact; until it ships, the Approver judges it from the diff and
   flags its absence as a finding (not an automatic
   `REQUEST_CHANGES`).

## Per-type criteria

The language overlay may add must-haves and risk factors to any type
below; apply core + overlay together.

### `feat:` — new feature

**Must-haves:**

- A linked GitHub issue (`Closes #N` or `Fixes #N` in the body).
- The implementation visibly addresses the user story from the issue
  body.
- New code paths have **meaningful tests** per *Test
  representativeness* — including the pyramid lens: logic under unit
  test, crossed boundaries under integration test.
- Public API surface changes are documented in `CONTRACTS.md` (when
  the repo ships `contracts/`).

**Risk factors:**

- Public API surface increased without a `CONTRACTS.md` update.
- Tests look like coverage-farming (assertions only on mock return
  values, no behaviour assertions).
- No regression test for the specific user-story claim.
- Net added lines > 500 (large features are harder to evaluate).

### `fix:` — bug fix

**Must-haves:**

- A regression test that fails without the fix and passes with it.
- The fix addresses the root cause, not the symptom.
- Linked to the original bug report when one exists.

**Risk factors:**

- Diff narrows error handling without addressing the underlying bug
  (catch-and-swallow pattern).
- Test added doesn't actually exercise the failure mode.
- The fix touches more files than the bug's scope would suggest.

### `refactor:` — behaviour-preserving restructure

**Must-haves:**

- No public API change (semver-compatible). **How** this is verified is
  language-specific — the overlay names the gate artifact to read when
  one exists, or the manual verification procedure when the gate isn't
  built. Either way: even if a gate's version-bump bypass allowed a
  break through, `refactor:` is non-negotiably no-break — flag a
  `REQUEST_CHANGES`.
- Coverage on touched files holds (drop > 0.5% is a finding).
- Diff is atomic — one refactoring intent per PR.

**Risk factors:**

- Public API changed without `refactor!:` and a matching version bump.
- Multiple unrelated refactoring intents batched together.
- Touched code's tests not re-run (CI ran but the touched module's
  tests are filtered out by a path-include rule).

### `chore(deps):` — patch / minor dependency bump

**Must-haves:**

- Dependabot-opened OR the title cites the package being bumped.
- Release-notes link present (PR body or comment).
- No `BREAKING CHANGE` markers in the release notes.

**Risk factors:**

- Patch / minor from a library with a known history of shipping
  behaviour changes under patch/minor tags (the overlay may name
  ecosystem-specific offenders).

### `chore(deps-major):` — major dependency bump

**Must-haves:**

- Migration notes from the release notes are visibly addressed in the
  diff or PR body.
- The language's major-upgrade agent ran successfully (when dispatched
  via `/development:maintenance` — confirm by checking the PR body for
  the agent's structured commit-body markers; the overlay names the
  agent).
- All consumers of the upgraded library's public API in this repo are
  updated.

**Risk factors:**

- Migration sections in the release notes that aren't visibly handled.
- The upgrade itself introduces a breaking change on *our* exposed
  contract (per `CONTRACTS.md`).

### `chore(runtime):` — language runtime or Docker base-image bump

The runtime surfaces (interpreter/toolchain pins, base images) and the
runtime-upgrade agent's availability are language-specific — **the
overlay is authoritative for this type.** Core invariants that always
apply:

**Must-haves:**

- The runtime version is **consistent across every surface that pins
  it** (the overlay enumerates them — e.g. build-config toolchain vs
  Dockerfile FROM line).
- When the language's runtime-upgrade agent exists, its structured
  commit body (Runtime / Cascade upgrades / Code adaptations) matches
  the diff; every cascade bump has a published version available.

**Risk factors:**

- Adaptations claimed in the commit body that don't appear in the diff.
- New runtime version is unreleased (pre-release / early-access tag).
- When no runtime-upgrade agent exists yet, there is no structured
  commit body to cross-check — the overlay says whether confidence is
  capped and human review expected.

### `security:` — CVE / finding fix

**Must-haves:**

- A test that demonstrates the unsafe input no longer succeeds.
- Linked to the CVE, Snyk alert ID, or CodeQL alert.

**Risk factors:**

- Fix narrows the input rather than fixing the unsafe path.
- Multiple unrelated security fixes batched (each should be its own
  PR for clean audit trails).

### `docs:` — documentation only

**Must-haves:**

- Claims in the diff cross-check against current code (file names,
  symbol names, behaviour descriptions).

**Risk factors:**

- Code examples that wouldn't run or compile (the Approver can attempt
  to syntax-check them but doesn't execute or build).
- Documentation referencing removed or renamed symbols.

### `test:` — tests only

**Must-haves:**

- *Test representativeness* applies in full — assertions bind to
  behaviour, no filler, no mocking the unit under test (the overlay
  lists the language's filler idioms).

**Risk factors:**

- Assertions only against mock return values.
- Test names that promise behaviour the assertions don't verify.

### `ci:` / `build:` — workflows / build config

**Must-haves:**

- No required gate weakened (coverage floor, branch-protection contexts,
  Sonar gate conditions).
- New jobs use least-privilege permissions (`permissions: contents: read`
  at the workflow level; widen only per job).

**Risk factors:**

- A required check was renamed without updating branch protection.
- New workflow runs on `pull_request_target` without `actions/checkout`
  pinning to `github.event.pull_request.head.sha` (RCE vector).

### `chore:` — cleanup / maintenance

**Must-haves:**

- Dead-code removal verified **including dynamic references** — the
  overlay lists the language's dynamic-dispatch idioms (reflection,
  string-keyed registries, plugin entry points).

**Risk factors:**

- Removed code is referenced by one of those dynamic mechanisms.

### `revert:` — clean revert

**Must-haves:**

- Clean revert (no follow-up commits with unrelated fixups).
- Dependents since the original commit have been considered.

**Risk factors:**

- Other PRs depend on the reverted commit.
- The original commit was a bugfix and the bug is now back.

### `hotfix:` — emergency

**Always `REQUEST_CHANGES` with "human review required".** Hotfixes are
emergencies; the Approver's confidence model is not calibrated for them
and a human must make the call. The Approver still produces a finding
list, so the human reviewer benefits from the analysis even though the
verdict is fixed.

## API stability (when `CONTRACTS.md` is present)

When the producer repo ships contract artifacts (per the polyrepo design
in `ARCHITECTURE.md` of the plugin family — issue #174), the Approver
applies these per-type additions:

- `feat:`, `fix:` → diff against `contracts/` must be non-breaking, UNLESS
  the PR title declares `feat!:` / `fix!:` AND a matching version bump
  is in the diff (the overlay names where this repo's version lives).
- `refactor:` → `contracts/` MUST NOT change.
- `chore(deps-major):` → consider whether the upgrade itself forces a
  contract change downstream.

Detector tools per surface. The **library public-API** surface is
language-specific — the overlay says which detector applies and whether
its gate is built (and therefore whether there is a machine-readable
findings artifact to read, or the Approver verifies no-public-break
manually). The language-independent surfaces are specified here but not
yet wired — they'll be added as #174 covers each:

| Surface | Marker | Detector | Source for the Approver |
|---|---|---|---|
| REST / OpenAPI | `contracts/openapi.yaml` | `oasdiff` | TBD |
| gRPC / Protobuf | `*.proto` under `contracts/` | `buf breaking` | TBD |
| GraphQL | `*.graphql` under `contracts/` | `graphql-inspector` | TBD |
| AsyncAPI / event schemas | `contracts/asyncapi.yaml` | `asyncapi-diff` | TBD |
| **Library public API** | *(per the language overlay)* | *(per the overlay)* | *(per the overlay)* |
| CLI parameters | `contracts/cli-help.txt` snapshot | line-diff | TBD |
| Env vars / config / webhooks | JSON Schema files under `contracts/` | JSON Schema diff | TBD |

A breaking diff without the matching version-bump signal is automatic
`REQUEST_CHANGES`. The bypass logic a gate uses (`!` in title + major
version bump) is **not** the same as the Approver's per-type rules —
`refactor:` PRs that a gate allowed through with `!` still trigger
`REQUEST_CHANGES` here, because refactors are non-negotiably
behaviour-preserving.

## Review dossier (when present)

A PR opened through the autonomous review loop carries a **Review dossier** in
its body — a hidden `<!-- review-dossier: {…} -->` JSON block that records what
the pre-merge reviewer panel already found and fixed. When it is present, parse
it and fold it into the risk register (it is **context, never a rubber stamp** —
CI and the per-type criteria still decide):

- **A `clean` dimension lowers that lens's residual-risk weight.** If
  `dimensions.<lens>.clean == true`, the panel reviewed that lens (bugs,
  security, performance, code_quality, tests — and swift6_compliance on Swift)
  and found nothing on the changed surface; treat a fresh risk you would
  otherwise raise in that lens as lower-probability, and do not manufacture one
  just to fill the register.
- **A non-clean dimension is where to look hardest** — the panel found something
  there; confirm the fix rather than assume it.
- **`waived_low` is context, not a finding.** Those are Low suggestions the loop
  logged and deliberately did not act on; never turn a waived-Low into a
  `REQUEST_CHANGES` on its own.

A PR **without** a dossier (e.g. `--no-review`, or a human-authored PR) is judged
exactly as before — the dossier only ever *reduces* unnecessary friction, never
adds a gate.

## Confidence calibration

Confidence is `HIGH` / `MEDIUM` / `LOW`. The Approver only `APPROVE`s
at `HIGH`. `MEDIUM` → `COMMENT`-with-reservations ("would approve if X
verified by a human"); `LOW` → `REQUEST_CHANGES`.

Adjustments to a default `HIGH`:

- Each unmet baseline criterion: drop one level.
- Each unmet per-type must-have: drop one level.
- Each risk factor matched: drop half a level (rounds down).
- Coverage drop on touched files > 0.5%: drop one level.
- PR has > 30 changed files OR > 1000 net added lines: cap at `MEDIUM`
  (large diffs are harder to evaluate reliably).
- Conventional-commit type cannot be determined: cap at `LOW`.
- Any additional caps the language overlay declares (e.g. a type whose
  supporting agent hasn't shipped yet).

**Special case:** PRs from `claude-maintenance[bot]` with green CI and
zero new tool findings start at `HIGH` and can `APPROVE` directly — the
maintenance pipeline already ran tests + tool-specific verification, and
the bot is on the allowlist by default.

## REQUEST_CHANGES feedback loop (JSON block)

Every `REQUEST_CHANGES` verdict appends a **hidden HTML-comment JSON
block** to the review body. The `/development:maintenance` skill reads
this on the next maintenance run and dispatches the right triage agents
on the PR branch.

Schema (the `suggested_agent` vocabulary — which triage agents exist
for this language — is listed in the overlay):

```json
{
  "approver_version": "v1",
  "verdict": "REQUEST_CHANGES",
  "confidence": "LOW",
  "type_detected": "feat",
  "findings": [
    {
      "category": "test_quality | api_stability | coverage | risk | type_ambiguity | baseline | …",
      "title": "Short headline",
      "detail": "Multi-line markdown explanation",
      "suggested_agent": "<an agent from the overlay's vocabulary> | null",
      "file": "path/to/file",
      "line": 42
    }
  ]
}
```

The block is wrapped in an HTML comment so it doesn't render in the
GitHub UI:

```markdown
<!-- claude-approver:findings
{ … the JSON above … }
-->
```

`/development:maintenance` parses the most recent Approver review's
hidden block, maps each finding's `suggested_agent` to the right triage
agent, dispatches them on the PR branch, pushes the fixes, and the
Approver re-runs on the next `check_suite: completed`.

# Language overlay — Python

Where the core policy says "per the language overlay", this part is the
reference. It adds Python fluency to the core's criteria; it never
relaxes them.

## Fallback diff-heuristic paths

- **Test paths:** `tests/`, `**/test_*.py`, `**/*_test.py`,
  `conftest.py`.
- **Build-config paths (→ `ci`):** the `pyproject.toml` build-system
  table.
- **Dependency manifests (→ `chore_deps`):** `pyproject.toml`
  `[project.dependencies]` / `[project.optional-dependencies]`,
  `requirements*.txt`, `poetry.lock`, `uv.lock`, `Pipfile.lock`.
- **Runtime pins (→ `chore_runtime`):** a `Dockerfile` FROM line as the
  only non-manifest diff.

## Test idioms (for the core's *Test representativeness*)

Coverage-padding patterns to flag: `assert True` / `assert 1` / `pass`
as the only body; assertions only on mock return values
(`mock.return_value = X; assert call() == X`); tests that
`unittest.mock` the very unit they test; test names like
`test_handles_empty_list` with no empty-list assertion.

## Per-type additions

### `chore(deps-major):`

- The major-upgrade agent is **`python-major-upgrade`**.

### `chore(deps):`

- Known patch/minor breakage offenders in this ecosystem: e.g.
  `pydantic` ≤ 2.0.

### `chore(runtime):` — Python or Docker base-image bump

- The runtime-upgrade agent is **`python-runtime-upgrade`**; its
  structured commit body (Runtime / Cascade upgrades / Code
  adaptations) is the cross-check, and every cascade bump must have a
  version available on PyPI.
- Runtime surfaces that must agree: `pyproject.toml`
  `requires-python` and the Dockerfile FROM line.

### `chore:` — dead-code dynamic references

Verify removal against `getattr`, `globals()`, dynamic `importlib`
calls, plugin entry points in `pyproject.toml`, and string-keyed
registries.

## API stability — library public-API surface

| Surface | Marker | Detector | Source for the Approver |
|---|---|---|---|
| **Library public API (Python)** | `[project]` in `pyproject.toml` | `griffe` (via `.github/scripts/check-api-stability.py`) | `griffe-findings.json` from the `api-stability` workflow |

The gate **is built** for Python: the `api-stability` workflow is
bootstrap-generated when `pyproject.toml [project]` is present (SKILL
Step 3f), and the Approver reads its `griffe-findings.json` artifact
directly for the core's `refactor:` no-public-break must-have. This
repo's version lives in `pyproject.toml` `[project] version` (the
core's version-bump signal).
